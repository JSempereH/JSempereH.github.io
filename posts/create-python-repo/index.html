How to create a simple Python project

Jul 20, 2025
5 minute read

In this post I’d like to write down the steps to make the simplest Python project you could make. Mainly focused for research projects (e.g. people in academia). Therefore, there will be no pipelines (like Github Actions or Jenkins).

Version control

Let’s say you have created your local directory: mkdir my-project. First, you need to init the version control. As of today, the most used tool is git.

1
2

cd my-project
git init

You are probably using Github, Gitlab for Codeberg for hosting your repo. Let’s say you are using Github, and you have already created your repo there. Now you need to add the remote repository reference to your local Git:

1

git remote add origin https://github.com/your-user/your-project.git

This command is setting the name “origin” to that remote. You can get the URL of the remote Git repository directly on the web:

Another way of doing this is directly creating the Github repo and then clone it locally:

1

git clone https://github.com/your-user/your-project.git

In case it is a private repo, you will need to log in.

You will need to create a .gitignore file. This will tell Git which file it shouldn’t track. For example:

# Ignore Python cache
__pycache__/
*.pyc

# Ignore virtual environment
.venv/

# Ignore logs
*.log

# Ignore IDE files
.vscode/

You can indicate whole directories (e.g. .venv/) or specific files, like all the files that end with .log: *.log.

Package manager

I will use uv for this post, since I think it’s one of the best tools. You can also use pip, poetry, pyenv, etc.

In case you don’t have it yet, you can install it with:

1

curl -LsSf https://astral.sh/uv/install.sh | sh

Let’s say you are already in your repo directory. Init uv with:

1

uv init

This will create the pyproject.toml, uv.lock, main.py (you will probably delete it), README.md, .python-version and a .gitignore (if you don’t have it yet).

  • pyproject.toml: this is a configuration file. It contains info about the project name, version, authors, dependencies, dev-dependencies (packages only needed for development), etc. For example:
1
2
3
4
5
6
7
8

[project]
name = "my_project"
version = "0.1.0"
requires-python = ">=3.12"
dependencies = ["numpy>=2.3.1"]

[tool.uv]
dev-dependencies = ["pytest", "ruff"]
  • uv.lock: this lockfile states exact versions of all dependencies, including transitive ones (those required by your direct dependencies).

If you want to use a new dependency in your code (e.g. numpy) you can use:

1

uv add numpy

This will update pyproject.toml and uv.lock. If you want to install a dev-dependency (e.g. pytest), you use:

1

uv add --dev pytest

You can remove a package with uv remove. In case another collaborator is working in your project and they update the packages, you can sync your virtual environment with the dependencies listed in pyproject.toml and uv.lock with uv sync.

If you want to install a library locally (not changing the project’s dependencies), use uv pip. For example, if you want to create a notebook that uses your python module, but don’t want to add ipykernel to the list oof dependencies, you may use:

1

uv pip install ipykernel

As a rule of thumb, try to minimize the number of dependencies in your project. Maintaining a project means updating the dependencies and handling the possible issues. The fewer the dependencies, the better.

Before you start writing your code, it’s better to set up some Python tools that format and correct your code.

Linter and formatter

A linter will check your code for potential errors, bad practices, styles issues and bugs. And a formatter will reformat your code automatically, so you don’t have to worry about spaces or line breaks.

Ruff can be used for both linting and formatting, you may install it as a dev-package:

1

uv add --dev ruff

Pre-commit

Instead of running ruff while you are developing the code, you could run it every time you try to commit some code using pre-commit. This way, it will format the the code and warn you if changes must be made. Install it also as a dev-package:

1

uv add --dev pre-commit

Now you need to create a .pre-commit-config.yaml file in the root of your project. The following config tell pre-commit to run ruff for linting and formatting:

1
2
3
4
5
6

repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.8  # Use the latest tag from https://github.com/astral-sh/ruff-pre-commit/releases
    hooks:
      - id: ruff
      - id: ruff-format

Here you can check Ruff’s default configuration.

Finally, install the Git hook:

1

pre-commit install

Now, every time you run git commit, Ruff will check and/or format your staged files automatically. If it requeries a manual change, you won’t be able to commit until you fix the code.

You can ignore pre-commit using -nm instead of -m, for example:

1

git commit -nm "This commit hasn't run pre-commit :("

You may also apply the hook to all files (not just the staged ones) using:

1

pre-commit run --all-files
